# 文档驱动全栈AI自动化开发工作流

## 核心定位

构建**技术栈智能调研、文档全链路驱动、变更自动同步、全节点可校验、全版本可追溯**的全栈应用自动化开发体系，解决AI开发中技术选型盲目、需求跑偏、文档代码脱节、模块混乱、迭代不可控等问题。

## 项目类型判断（流程入口）

启动时必须先判断项目类型：

| 项目类型 | 判断条件 | 执行路径 |
|----------|----------|----------|
| 新项目 | 当前目录无项目代码/无docs文档体系 | 从阶段-1开始 |
| 老项目 | 当前目录存在项目代码/存在docs文档体系 | 直接执行阶段0或后续阶段 |

---

## 全局强制铁律（违反任意一条直接终止）

1. **文档先行**：无文档不开发，无设计不编码
2. **计划先行**：开始开发代码前必须先生成实施计划（PLAN），计划存档至 `docs/plans/`，无计划不编码
3. **索引闭环**：每层级目录必须有 `index.md`，实现全链路可索引
4. **变更前置备份**：任何修改前必须先触发全量快照备份
5. **关联文档自动同步**：需求变更后必须基于依赖矩阵同步所有下游文档
6. **一致性校验**：每环节完成后必须AI自检，上一环节未通过校验禁止进入下一环节
7. **最小颗粒度**：单页面、单接口、单功能点对应独立文档与唯一标识
8. **边界锁定**：所有执行动作严格限定在文档约定范围内

## 阶段-1：技术栈调研与选型（新项目专属前置阶段）

### 执行目标

根据用户需求智能调研最适合的技术栈、架构模式、设计规范，输出完整的技术选型报告，为后续项目初始化提供锁定依据。

### 前置输入

用户原始需求描述、项目核心目标、预期规模、团队技术背景（可选）

### 标准执行步骤

#### 步骤1：用户需求解析与项目定位

1. 解析用户原始需求，明确项目类型：
   - Web应用（SPA/MPA/PWA）
   - API服务（RESTful/GraphQL/gRPC）
   - AI应用（AI Agent/RAG/LLM应用）
   - 移动应用（原生/混合/小程序）
   - 工具类/CLI工具
   - 行业解决方案

2. 明确项目规模等级：
   - 小型：单功能/工具类，开发周期<2周
   - 中型：多模块/SaaS产品，开发周期2-8周
   - 大型：企业级解决方案，开发周期>8周
   - 超大型：分布式系统，多团队协作

#### 步骤2：主语言选择交互

**必须通过 AskUserQuestion 工具让用户选择主语言方向**

| 选择项 | 子选项 | 适用场景 |
|--------|--------|----------|
| 前端主导 | JavaScript / TypeScript | Web应用、移动应用前端 |
| 后端主导 | Python / Java / JavaScript/TypeScript / Go / Rust / C# | API服务、微服务、AI应用后端 |
| AI工作流 | Python / JavaScript/TypeScript | AI Agent、RAG、LLM应用 |
| 全栈统一 | TypeScript（前后端统一） | 快速迭代、中小型项目 |
| 其他 | 用户自定义 | 特殊需求场景 |

#### 步骤5：架构模式与设计规范调研

**架构模式调研矩阵**

| 项目类型 | 推荐架构 | 说明 |
|----------|----------|------|
| 小型SPA | 单体分层 | 简单直接，快速交付 |
| 中型Web | 前后端分离 + 模块化 | 清晰边界，易于扩展 |
| 大型企业 | 微服务 + 领域驱动(DDD) | 高扩展，团队协作友好 |
| AI应用 | Agent编排 + RAG管道 | 工作流驱动，知识检索增强 |
| API服务 | RESTful + 版本控制 | 标准化接口，易于集成 |

**设计规范调研**

| 规范类别 | 推荐方案 |
|----------|----------|
| 代码风格 | ESLint + Prettier / Black / gofmt |
| 命名规范 | 文件：kebab-case，变量：camelCase，类：PascalCase |
| 目录规范 | 按功能/领域分层，src/[domain]/[layer] |
| API规范 | OpenAPI 3.0，版本化路由 /api/v1/ |
| 安全规范 | JWT/OAuth2，HTTPS强制，输入校验 |

## 阶段0：项目初始化与自动化底座搭建

### 执行目标

完成标准化骨架搭建、全局规范定义、AI执行边界锁定，同时搭建依赖矩阵和备份体系两大核心自动化底座。

**关键**：无论新项目还是老项目，阶段0完成后都必须生成/更新 `AGENTS.md`

### 前置输入

**新项目**：阶段-1输出的技术选型报告 + AGENTS.md（已生成）
**老项目**：需要先分析现有代码结构，推断技术栈和规范，再生成 AGENTS.md

### 标准执行步骤

#### 新项目执行流程

1. **定义全局规范**（基于阶段-1技术选型）：
   - 技术栈锁定：明确前后端框架、版本号、核心依赖
   - 编码规范：命名规则、注释规范、代码分层规则、格式化标准
   - 版本规则：语义化版本号规范、分支管理规则
   - 安全基线：鉴权规则、数据加密规则、接口安全规范
   - 文档规范：统一文档模板、命名规则、变更标记规范

2. 按标准目录结构搭建项目骨架

3. 生成根索引文档 `docs/index.md`

4. 搭建依赖矩阵 `doc-dependency-matrix.yaml`

5. 搭建自动化版本备份体系

6. 完成首次全量快照备份，生成初始版本v1.0.0

7. **确认 AGENTS.md 已正确生成**（阶段-1已完成）

#### 老项目执行流程（新增步骤）

1. **分析现有项目结构**：
   - 检查 `package.json` / `requirements.txt` / `go.mod` / `pom.xml` 等依赖文件
   - 分析目录结构，推断分层架构
   - 检查配置文件（ESLint、Prettier、tsconfig、pyproject.toml等）
   - 检查现有代码风格和命名规范

2. **推断技术栈**：
   | 分析项 | 检查文件/方式 |
   |--------|---------------|
   | 前端框架 | package.json 中的 dependencies |
   | 后端框架 | requirements.txt / go.mod / pom.xml |
   | 数据库 | 配置文件、ORM文件、migration文件 |
   | 语言版本 | tsconfig / pyproject.toml / go.mod |
   | 构建工具 | package.json scripts / Makefile |
   | 测试框架 | 测试文件、配置文件 |

3. **生成/更新 AGENTS.md**：
   基于分析结果，生成完整的 `AGENTS.md` 文件，包含：
   - 从现有代码推断的技术栈锁定信息
   - 现有的架构模式和分层结构
   - 现有的编码规范（从配置文件和代码风格推断）
   - 现有的API规范（从现有接口推断）
   - 现有的安全规范（从现有代码推断）
   - 文档驱动开发规范（新增）

   **如果项目已有 AGENTS.md，需对比差异并更新**

4. **搭建docs文档体系**：
   - 创建标准目录结构 `docs/`
   - 生成根索引文档 `docs/index.md`
   - 创建各子目录及其 `index.md`
   - **创建 `docs/plans/` 目录结构**：
     ```
     docs/plans/
     ├── index.md          # 计划索引文档
     ├── .history/         # 计划历史版本备份
     ├── active/           # 当前执行中的计划
     │   └── index.md
     └── archived/         # 已完成/已归档的计划
         └── index.md
     ```

5. **搭建依赖矩阵 `doc-dependency-matrix.yaml`**

6. **搭建自动化版本备份体系**

7. **完成首次全量快照备份，生成初始版本v1.0.0**

8. **更新根索引文档，记录AGENTS.md的生成记录**

### 强制输出物

1. 100%符合标准的项目目录结构（新项目）或补充docs体系（老项目）
2. 根索引文档、全局规范文档
3. 依赖矩阵文件
4. 完整的备份体系
5. `version.json`版本管控文件
6. 项目初始化全量快照备份
7. **AGENTS.md**（新项目确认已有，老项目新生成）

### 准出校验

- 目录100%符合标准、索引完整、依赖矩阵可解析、备份完整、规范无歧义
- **AGENTS.md 已生成且包含完整的技术栈锁定信息**
- 老项目：技术栈推断准确，与现有代码一致

## 阶段1：需求全链路标准化文档化

### 执行目标

将模糊需求转化为结构化、可量化、可追溯的标准化文档

### 标准执行步骤

1. 需求拆解：业务目标、核心功能、非功能需求、约束条件、异常场景、验收标准
2. 分配唯一需求ID（格式：REQ-模块编码-序号）
3. 生成顶层文档 `prd.md`、`srs.md`
4. 分模块拆解最小单元需求文档
5. 搭建需求域索引体系
6. 同步更新依赖矩阵
7. AI自检

### 准出校验

所有需求点有唯一ID、验收标准可量化、无业务矛盾、索引闭环

## 阶段3：分端模块化自动化开发

### 执行目标

基于校验通过的设计文档，按模块完成代码自动化生成。

**关键**：开发代码前必须先生成实施计划（PLAN），无计划不编码！

### 前置输入

阶段2所有子流程校验通过的设计文档、阶段0的全局编码规范、全链路依赖矩阵

### 标准执行步骤

#### 步骤1：生成实施计划（PLAN）- 强制前置步骤

**开发任何代码前，必须先生成实施计划并存档至 `docs/plans/`**

1. **计划文件命名规则**：
   - 格式：`PLAN-{类型}-{模块名}-{日期}-{序号}.md`
   - 示例：`PLAN-FE-user-module-20260409-001.md`（前端用户模块开发计划）
   - 示例：`PLAN-BE-auth-api-20260409-002.md`（后端认证接口开发计划）

2. **计划内容模板**（详见 [实施计划模板](references/plan-template.md)）：
   ```markdown
   # 实施计划：[计划名称]
   
   ## 一、计划基础信息
   - 计划ID：PLAN-{类型}-{模块}-{日期}-{序号}
   - 关联需求ID：REQ-XXX-XXX
   - 关联设计文档：[设计文档路径]
   - 执行范围：前端/后端/全栈
   - 预计工时：[预估时间]
   
   ## 二、开发任务拆解
   | 序号 | 任务名称 | 对应设计文档 | 输出文件 | 预计时间 |
   |------|----------|--------------|----------|----------|
   | 1 | [任务名] | [文档路径] | [文件路径] | [时间] |
   
   ## 三、技术实现要点
   [关键技术点和实现方案]
   
   ## 四、依赖关系
   [任务之间的依赖关系]
   
   ## 五、风险点与应对
   [潜在风险和应对措施]
   
   ## 六、验收标准
   [完成标准]
   ```

3. **计划存档**：
   - 新计划存入 `docs/plans/active/`
   - 更新 `docs/plans/index.md` 索引
   - 计划状态：pending（待执行）、in_progress（执行中）、completed（已完成）、cancelled（已取消）

4. **用户确认计划**：
   - 使用 AskUserQuestion 让用户确认计划是否可行
   - 确认后方可进入开发阶段

#### 步骤2：开发前前置校验

AI再次校验对应设计文档的有效性，确认文档已纳入索引、已通过校验、为最新版本，禁止使用未校验/旧版本文档；同时确认对应需求ID、接口ID/页面ID与依赖矩阵一致。

#### 步骤3：标记计划状态为 in_progress

更新计划文件，标记状态为执行中，记录开始时间。

#### 步骤4：模块化拆分开发

严格按照「先底层公共模块、后业务模块；先数据层、后服务层、再接口层；先组件开发、后页面开发」的顺序，分模块、分单元开发，每个最小单元对应唯一的设计文档，禁止跨模块开发。

#### 步骤5：前端开发执行规范

- 第一步：基于设计系统文档，开发全局样式、基础组件库，完成后校验与设计规范的一致性。
- 第二步：基于页面地图文档，搭建路由架构、权限控制框架、全局状态管理。
- 第三步：基于单页面设计文档，逐页面开发，每个页面对应独立的代码模块，代码开头必须标注对应设计文档的路径、需求ID、页面ID。
- 第四步：页面开发完成后，AI自检与设计文档的还原度、交互逻辑一致性、异常场景覆盖度。

#### 步骤6：后端开发执行规范

- 第一步：基于库表设计文档，生成数据库表结构、初始化SQL、数据访问层代码。
- 第二步：基于领域模型设计文档，开发领域服务、业务规则处理逻辑。
- 第三步：基于单接口设计文档，逐接口开发Controller层、服务层代码，每个接口代码开头必须标注对应接口文档的路径、接口ID、需求ID。
- 第四步：接口开发完成后，AI自检与接口设计文档的一致性，包括入参出参、业务逻辑、错误码、异常处理。

#### 步骤7：代码索引体系搭建

完善 `src/frontend/index.md`和 `src/backend/index.md`，记录每个代码模块的路径、对应设计文档路径、对应需求/接口ID、版本信息，实现代码与设计文档的双向追溯。

#### 步骤8：单元测试自动化生成

基于设计文档，为每个模块、每个接口、每个页面生成对应的单元测试用例，完成单元测试验证，核心逻辑覆盖率100%。

#### 步骤9：标记计划状态为 completed

更新计划文件，标记状态为已完成，记录完成时间、实际工时、输出文件清单。

#### 步骤10：计划归档与备份

- 将已完成的计划从 `docs/plans/active/` 移动到 `docs/plans/archived/`
- 保存历史版本至 `docs/plans/.history/`
- 更新 `docs/plans/index.md` 索引

### 强制输出物

1. **实施计划文档**：`docs/plans/active/PLAN-xxx.md`（执行中）或 `docs/plans/archived/PLAN-xxx.md`（已完成）
2. 全量可运行的前端/后端源代码，按模块拆分，符合全局编码规范
3. 代码模块两级索引体系：`src/frontend/index.md`、`src/backend/index.md`，以及子模块 `index.md`
4. 数据库初始化SQL脚本、数据迁移脚本
5. 每个模块/接口/页面的单元测试代码
6. 代码与设计文档双向追溯矩阵（嵌入代码索引文档）
7. 更新后的全链路依赖矩阵
8. 开发完成全量快照备份

### AI准出校验规则

- **计划已生成并存档**：无计划不得进入开发阶段
- 计划已通过用户确认
- 代码100%符合对应设计文档的要求，无超出文档范围的功能，无文档内需求遗漏
- 所有代码均已纳入索引体系，可通过索引文档追溯到对应的设计文档、需求ID，无游离代码
- 代码符合全局编码规范，命名统一、注释完整，关键逻辑标注了对应的文档依据
- 所有接口、页面的异常场景、边界条件均已按文档实现，无缺失
- 单元测试覆盖核心业务逻辑，测试通过率100%
- 项目可正常编译、启动，无语法错误、无依赖缺失
- 代码与设计文档双向追溯关系完整，已纳入依赖矩阵
- **计划状态已更新为 completed 并归档**

## AI自动化执行增强规范

1. **指令锁定**：明确「严格按照[文档路径]执行」
2. **分阶段锁定**：未通过准出校验禁止进入下一环节
3. **计划先行强制**：阶段3开发前必须先生成计划并存档，无计划不编码
4. **用户交互强制**：阶段-1必须使用 AskUserQuestion 进行技术栈确认
5. **自检自动化**：每环节完成后输出自检报告
6. **异常终止**：以下情况立即终止：
   - 新项目未完成技术栈调研直接进入开发
   - 技术栈版本未锁定（使用"最新版"等模糊描述）
   - 用户未确认技术选型方案
   - **开发代码前未生成实施计划**
   - **计划未存档至 `docs/plans/`**
   - **计划未通过用户确认**
